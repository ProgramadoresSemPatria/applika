## Creating a Migration

After modifying models in `app/domain/models.py`, generate a migration:

```bash
uv run task autorevision
```

This runs `alembic revision --autogenerate -m 'autogenerated'`. The generated file goes to `migrations/versions/`.

**Important:** Always review and edit the generated migration before applying it.

## Fixing Auto-Generated Migrations

Alembic auto-generation has a well-known issue: it uses `None` for constraint names. This causes problems when you later need to drop or alter constraints. Fix them manually.

### Constraint naming conventions

| Constraint type | Naming pattern                    | Example                           |
|----------------|-----------------------------------|-----------------------------------|
| Foreign key     | `fk_<table>_<column>`           | `fk_tags_user_id`               |
| Unique          | `uq_<table>_<column>`           | `uq_users_email`                |
| Index           | `ix_<table>_<column>`           | `ix_users_github_id`            |
| Primary key     | `pk_<table>`                     | `pk_tags`                        |
| Check           | `ck_<table>_<column>`           | `ck_applications_mode`          |

### Before (auto-generated)

```python
def upgrade() -> None:
    op.create_table(
        'tags',
        sa.Column('id', sa.Integer(), autoincrement=True, nullable=False),
        sa.Column('name', sa.String(length=100), nullable=False),
        sa.Column('user_id', sa.Integer(), nullable=False),
        sa.ForeignKeyConstraint(['user_id'], ['users.id'], ondelete='CASCADE'),
        sa.PrimaryKeyConstraint('id'),
    )
```

### After (fixed)

```python
def upgrade() -> None:
    op.create_table(
        'tags',
        sa.Column('id', sa.Integer(), autoincrement=True, nullable=False),
        sa.Column('name', sa.String(length=100), nullable=False),
        sa.Column('user_id', sa.Integer(), nullable=False),
        sa.ForeignKeyConstraint(
            ['user_id'], ['users.id'],
            name='fk_tags_user_id',
            ondelete='CASCADE',
        ),
        sa.PrimaryKeyConstraint('id', name='pk_tags'),
    )
```

### For add_column with foreign key

```python
# Auto-generated (bad — unnamed constraint)
op.create_foreign_key(None, 'applications', 'tags', ['tag_id'], ['id'])

# Fixed
op.create_foreign_key(
    'fk_applications_tag_id',
    'applications', 'tags',
    ['tag_id'], ['id'],
)
```

### For unique constraints and indexes

```python
# Fix unnamed unique constraint
op.create_unique_constraint('uq_tags_name', 'tags', ['name'])

# Fix unnamed index
op.create_index('ix_tags_user_id', 'tags', ['user_id'])
```

### Downgrade consistency

Make sure the `downgrade()` function references the same constraint names:

```python
def downgrade() -> None:
    op.drop_constraint('fk_applications_tag_id', 'applications', type_='foreignkey')
    op.drop_column('applications', 'tag_id')
    op.drop_table('tags')
```

### Revision message

Give the migration a descriptive message. Edit the docstring at the top:

```python
"""'tags' table created

Revision ID: abc123def456
...
"""
```

Or generate with a meaningful message from the start:

```bash
alembic revision --autogenerate -m "'tags' table created"
```

## Seed Data in Migrations

For reference data (platforms, step definitions, etc.), use `op.bulk_insert()`:

```python
from datetime import datetime, timezone

def upgrade() -> None:
    now = datetime.now(timezone.utc)
    table = sa.table(
        'tags',
        sa.column('name', sa.String),
        sa.column('color', sa.String),
        sa.column('created_at', sa.DateTime(timezone=True)),
    )
    op.bulk_insert(table, [
        {'name': 'Frontend', 'color': '#61dafb', 'created_at': now},
        {'name': 'Backend', 'color': '#3776ab', 'created_at': now},
    ])
```

## Migration Commands

```bash
uv run task autorevision    # Generate migration from model changes
uv run task auhead          # Apply all pending migrations (upgrade head)
uv run task adbase          # Rollback all migrations (downgrade base)
```

## Test Setup

Tests use `pytest` with `testcontainers` for a real PostgreSQL instance. The test infrastructure lives in `app/tests/conftest.py`.

### Key fixtures

- `db_container` (session-scoped): spins up a PostgreSQL 14 container once per test session
- `async_engine`: creates tables from scratch (drop_all + create_all) per test
- `db_session`: creates a session with seeded base data
- `async_client`: `httpx.AsyncClient` with overridden dependencies for the test DB and a mock current user

### How dependency override works

```python
main_app.dependency_overrides[get_session] = override_get_session
main_app.dependency_overrides[get_current_user] = override_get_current_user
```

This replaces the real DB session with one pointing at the test container, and replaces the auth dependency with a hardcoded test user.

## Writing Integration Tests

Tests go in `app/tests/integration/test_<feature>.py`.

```python
import pytest
from httpx import AsyncClient


@pytest.mark.asyncio
async def test_create_tag(async_client: AsyncClient):
    # Arrange
    payload = {
        'name': 'Frontend',
        'color': '#61dafb',
    }

    # Act
    response = await async_client.post('/tags', json=payload)

    # Assert
    assert response.status_code == 201
    data = response.json()
    assert data['name'] == 'Frontend'
    assert data['color'] == '#61dafb'
    assert 'id' in data
    assert 'created_at' in data


@pytest.mark.asyncio
async def test_list_tags(async_client: AsyncClient):
    # Arrange — create a tag first
    await async_client.post('/tags', json={
        'name': 'Backend',
        'color': '#3776ab',
    })

    # Act
    response = await async_client.get('/tags')

    # Assert
    assert response.status_code == 200
    data = response.json()
    assert len(data) >= 1


@pytest.mark.asyncio
async def test_delete_tag(async_client: AsyncClient):
    # Arrange
    create_resp = await async_client.post('/tags', json={
        'name': 'To Delete',
        'color': '#ff0000',
    })
    tag_id = create_resp.json()['id']

    # Act
    response = await async_client.delete(f'/tags/{tag_id}')

    # Assert
    assert response.status_code == 204
```

### Test conventions

- Arrange-Act-Assert pattern
- Use `async_client` for full API integration tests (tests the entire stack)
- Use `db_session` when you need to test repository logic directly
- `asyncio_mode = "auto"` is set in pyproject.toml so `@pytest.mark.asyncio` is technically optional but kept for clarity

## Test Data Seeding

Base test data is defined in `app/tests/base_db_setup.py`:

```python
def base_data() -> BaseDataType:
    return {
        'user': UserModel(id=1, github_id=1, username='testuser', email='test@user.com'),
        'plat_linkedin': PlatformModel(id=1, name='Linkedin', url='https://www.linkedin.com/'),
        'fb_denied': FeedbackDefinitionModel(id=1, name='Denied', color='#a80000'),
    }
```

To add seed data for a new feature, add entries to this dictionary. The `db_session` fixture calls `session.add_all(base_data().values())` before each test.

### Running tests

```bash
uv run task pytest    # Run all tests with coverage
```
